## Prog10 Tinker Overview

This submission implements a dual-issue pipelined Tinker core in [`tinker.sv`](/u/lanthony/cs429/CS-429-Comp-Arch-Prog10/tinker.sv) with the main CPU module:

```systemverilog
module tinker_core (
    input clk,
    input reset,
    output logic hlt
);
```

The core uses a Tomasulo-style structure with:

- A register alias table and 64-entry physical register file for register renaming
- A reorder buffer for in-order commit
- Reservation stations for integer/branch and floating-point work
- Two pipelined integer ALUs and two pipelined FPUs
- A load/store queue with store-at-commit behavior and store-to-load forwarding
- A simple dynamic branch predictor with BTB-based target prediction

`hlt` is asserted only when the oldest `PRIV 0` instruction commits from the reorder buffer.

## File Layout

- `tinker.sv`: top-level include file for the design
- `hdl/`: module definitions for the core, fetch shell, decoder, reg file, memory, and ALU/FPU units
- `test/`: simulation testbenches
- `sim/`: waveform output generated by the testbenches
- `vvp/`: optional compiled simulator outputs

## Compile And Run

All commands below were verified with Icarus Verilog 12.0.

### Comprehensive end-to-end test

```bash
iverilog -g2012 -o vvp/comprehensive_run tinker.sv test/tb_comprehensive.sv
vvp vvp/comprehensive_run
```

This produces `sim/comprehensive.vcd`.

### Multicycle feature test

```bash
iverilog -g2012 -o vvp/new_run tinker.sv test/tb_new.sv
vvp vvp/new_run
```

This exercises halt, integer ops, load/store behavior, branching, call/return, and floating-point execution.

### Floating-point focused test

```bash
iverilog -g2012 -o vvp/fp_run tinker.sv test/tb_fp_only.sv
vvp vvp/fp_run
```

This produces `sim/fp_only.vcd`.

### Integer smoke test

```bash
iverilog -g2012 -o vvp/and_run tinker.sv test/tb_and_only.sv
vvp vvp/and_run
```

This produces `sim/and_only.vcd`.

## Notes

- The architectural register file remains visible as `reg_file`, while the renamed physical register state is maintained internally by `tinker_core`.
- Stores are written to memory only when they reach the head of the reorder buffer and commit.
- The first floating-point unit instance is named `fpu` to match the assignment requirement.
